  1. ¿Que es git y para que sirve?
    • Es un sistema de control de versiones
    • Trabajo en equipo
    • Listado y fecha de cambios(commit)
    • Volver atrás en el tiempo restablecer cambios
    • Ramas de desarrollo
    • Mezclar cambios entre compañeros
  2. ¿Como instalar Git?
    • git-scm.com - windows
    • CYGWIN - virtual windows
    • Mac Build - Mac Version
    • apt-get install git - Linux
    • aptitude install git - Debian
  3. Configurar usuario
    • git --version
    • git config --global user.name "Andrés Bonfil" (nombre de usuario)
    • git config --global user.email "<andresbonfil@gmail.com>" (correo electronico)
    • git config --list (para ver si ya los tengo configurados)
  4. Ayuda en git
    • git help init - nos muestra la ayuda del comando init
  5. Crear un repositorio
    • Dentro de la carpeta del proyecto ejecutamos git init
    • ls -lha para ver el repositorio(carpeta) oculto .git
  6. Clonar repositorios
    • Desde repositorio local o desde la nuve
    • Desde una carpeta superior hacemos git clone ./05-miproyecto proyectoclonado - como no se ha commiteado nada aparece de vacio - pero aparece la rama maste o main asi como la carpeta oculta .git(elrepositorio) ls -lha (de esta manera clonamos de manera local)
    • git clone https://github.com/andresbonfil/andresbonfil.github.io.git proyectoNube nos clona desde github en una carpeta proyectonube
  7. Espacio de Trabajo (el stage)
    • git status - muestra el estado del stage en que rama nos encontramos
    • git add images/ - se añade la carpeta imagenes al stage con todo su contenido
    • git add index.html - añade el archivo index.html
    • git add . - añade absolutamente todo al stage
    • git rm --cached pagina1.html quita del stage la pagina1.html
  8. Status, add y diff
    • A este punto hacemos cambios que ya teniamos estageados todos los archivos y hacemos git status, nos aparecera que esos cambios no seran comiteados hasta que los estagiemos de nuevo // estageamos solo index y pagina uno no queremos los cambios git checkout pagina1.html nos devuelve a antes de los cambios
    • git diff pagina1.html muestra los cambios que ha sufrido el archivo
  9. Gitignore
    • creamos en la raiz del proyecto el archivo .gitignore (touch .gitignore)
    • podemos editar el archivo .gitignore con el comando nano .gitignore (dentro escribimos los archivos.extension que no queremos darles seguimiento y por ende stagear o comitear) el .gitignore si se puede estagear y comitear pero desde que se crea ya deja de trackear archivos ignorados en el archivo
    • Nomas al agregar el archivo conexion.php al archivo git ignore no importa si esta en subcarpeta, siempre va ser ignorado
    • Si agregamos al archivo gitignore la linea config/ la carpeta config y lo que contiene no va ser tomado en cuenta, donde quiera que la carpeta se encuentre
  10. Commit y versiones
    • git commit -m "Contiene todos los archivos iniciales"
    • git log --oneline (En una sola linea)
    • git log (Mas detallado)
    • git commit -am "cambios en los textos de la pagina" (stagea y comitea de un tiron)
    • git diff 945f188 f630f89 (compara el utltimo con el anterior[primero se pone el mas reciente y luego se pone el anterior si no lo hace todo al revéz])
  11. Reset y checkout
    • git reset HEAD index.html (desestagean cambios)
    • git reset . (desestagean cambios)
    • git checkout index.html (los cambios se deshacen) [no sirvió]
  12. Deshacer Commit
    • git reset --hard HEAD~0 (Este no borra nada ya que se queda en el HEAD)
    • git reset --hard HEAD~1 (Este borra EL ULTIMO)
    • git reset --hard HEAD~2 (Este borra LOS DOS ULTIMO y sí sucesivamente)
    • git reset --hard 72dca06 (te deja hasta ese commit)
    • git reset --hard 72dca06~1 (te deja hasta el comit anterior de ese commit)
    • git reset HEAD~1 (Esto deshace el ultimo commits pero te dejar en los archivos desestajeados por si los quiere utilizar)
  13. Volver a un commit anterior
    • git checkout <72dca06> (nos lleava para el commit especifico y se puede volver)
    • si hacemos un nuevo commit desde un commit anterior cuando volamos al ultimo o mas reciente commit nos advierte se perderá nos sugiere hacer le una nueva rama a ese commit que va ser olvidado
    • git branch <NOMBRE-RAMA> e11a530
    • git branch <NOMBRE-RAMA> y nos cambia de rama para seguir otra ruta
  14. Buscar Commits (Si tuvieramos miles de commits)
    • git log --grep=iniciales
    • commit 945f18888167a1197fda16a86279a814c0061838 Author: Andres Bonfil Date: Mon May 26 18:53:02 2025 -0600
    • puedo volver hacer cambios y luego volver al checkout master
    • git log -s 'titulo del h1 o algo dentro del commit' - nos devuelve commit relacionados
  15. Ramas y Merges (nos permite ramificar el proyecto para desarrollar en paralelo diferentes funcionalidades)
    • git branch - lista las ramas
    • git branch list - crea la rama list (si hacemos git branch ya nos aparece en la lista)
    • git ceckout <Nombre-Rama> nos cambia entre ramas (si hacemos cambios en una rama no apareceran en otras ramas)
    • git checkout master me devuelve a la rama master
    • git branch -d list (no borra si tiene cambios que no se han mesclado, si no tiene cambios si se borra)
    • git branch -D list (si borra aunque tenga cambios que no se hallan mezclado)
    • git merge contacto (desde la rama master donde no tengo la funcionalidad contacto)
    • git checkout pruebas y desde pruebas me quiero traer todo lo que tiene master y actualizar la rama de pruebas git merge master (y nos traer todo contacto y lo de master)
    • git help merge - muestra mas opciones
    • git help branch - mas opciones
  16. Stash
    • Si estubiera trabajando en la rama pruebas y mi jefe me dice oie urge cambiar la pagina de contacto, si directamente me cambio sin estagear o estageado lo que estoy haciendo se lo lleva a la rama de contacto
    • para no llevarte lo que estas trabajando y sin perderlos para eso stash
    • git stash (guarda en memoria temporal)
    • git stash pop devuelve
    • git stash clear LIMPIA EL STASH
  17. Cherry pick
    • Supongamos que terminamos una funcionalidad en una rama ej pruebas
    • Nos piden que esa funcionalida ya terminada la van a usar en produccion
    • solo queremos lo terminado no nos interesa tu historial de cambios
    • hacemos un cherry pick
    • git checkout master
    • git cherry-pick <hash> (del git log de la otra rama)
    • es un sustituto del merge pero mas especifico
  18. Como resolver conflictos
    • Si se hace un merge y dos usuarios tocaron la misma linea de codigo te va a pedir con cual te quieres quedar con el actual de tu rama o el entrante que te entra
    • Una vez lo decides solo vuelves stagear y comitear
  19. Repositorios remotos y github
    • hacer repositorio en github
    • git remote add origin https://github.com/usuario/repo1.git
    • git push -u origin master
    • push para subir
    • pull para bajar
    • desde remoto creo una rama usuario y creo un archivo config.yml
    • git fetch --all (nos baja todas las ramas en remoto)
    • git checkout usuarios
    • git fecht --all
    • literal todo lo mas basico
